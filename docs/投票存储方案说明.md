# 竞品投票：数据存哪里？

## 结论一览

| 方案 | 是否推荐 | 说明 |
|------|----------|------|
| **Supabase 云表** | ✅ 最推荐（配合 GitHub Pages） | 真正的数据库表，所有人打开 Pages 链接即可同步 |
| **自建后端 vote-server** | ✅ 局域网/自建服务器 | 项目已自带，数据在 `data/comp-votes.json` |
| **Render 等部署 vote-server** | ✅ 可选 | 有公网 API，在 `comp-vote-config.js` 填 `apiBase` |
| **Git 仓库当数据库** | ❌ 不推荐 | Pages 只能读不能写；用 API 写 Git 需密钥，无法实时、易冲突 |

Git 适合放**代码和静态页**，不适合当**多人实时投票**的写入型数据库。

---

## 方案 A：Supabase 云表（推荐）

适合：所有人只打开 **GitHub Pages** 地址，不在同一 WiFi。

### 1. 创建 Supabase 项目

1. 打开 <https://supabase.com> 注册并 **New project**
2. 进入 **SQL Editor**，执行仓库内文件：  
   `supabase/comp-votes.sql`

### 2. 获取连接信息

**Project Settings → API**：

- **Project URL** → 填入 `supabaseUrl`
- **anon public** key → 填入 `supabaseAnonKey`

### 3. 修改 `comp-vote-config.js`

```javascript
window.HX_VOTE_CONFIG = {
  storage: "supabase",
  supabaseUrl: "https://xxxx.supabase.co",
  supabaseAnonKey: "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  supabaseTable: "comp_votes",
  pollIntervalMs: 2000,
  adminToken: "hx-vote-admin",
  apiBase: "",
  staticHosts: ["github.io", "githubusercontent.com"]
};
```

### 4. 推送后使用

所有人访问同一 Pages 链接 → **竞品分析 → 5. 竞品偏好投票**  
同步状态应显示：**已连接 · Supabase 云同步**

在 Supabase **Table Editor** 中可看到 `comp_votes` 表里的每一票。

## 方案 B：局域网 vote-server（无需注册云账号）

```bash
npm run vote-server
```

所有人打开 `http://局域网IP:3456#comp-vote`。

---

## 方案 C：云主机部署 vote-server

1. 将仓库连接到 [Render](https://render.com)，使用根目录 `render.yaml` 部署  
2. 得到地址如 `https://hx-comp-vote.onrender.com`  
3. 在 `comp-vote-config.js` 设置：

```javascript
storage: "http",
apiBase: "https://hx-comp-vote.onrender.com/api"
```

> 免费实例可能休眠；重要会议建议用 Supabase 或付费实例。

---

## 为什么不用「Git 里建表」？

- GitHub Pages **没有服务器**，浏览器**不能直接**往仓库里的 JSON/CSV 写数据。  
- 若用 GitHub API 每次投票 commit 一次：慢、易冲突、需把 Token 放在前端（极不安全）。  
- **Supabase 表** = 真正的在线数据库，和「在 git 里放一张表」目的一样，但适合实时读写。

如需把结果**归档进 Git**，可在会后用页面上的 **导出投票 JSON** 手动提交到仓库。
